# Phase 6 report — Code generation

Built `core/codegen`: deterministic emission of a complete nkMatlib SystemVerilog
module from an audited DAG (CG-1/CG-4) — `fixedp` port `g`, `_0` inputs / `_N` outputs,
one operator instance per node with `i_<module>_<signal>_<stage>` naming (stage =
output-ready cycle), constants via `TOFXD`, and **all** matching delays computed from
the cost model: operand alignment pipes (`pipe #(.DELAY(n))`, deduplicated per
signal/stage), early-output alignment to the final stage, and the reset `valid` chain.
The linter was extended to honor `.DELAY` parameter overrides on pipe/valid instances
so generated code is verifiable by the same cycle model. CG-2 is enforced as a unit
test: every codegen golden file lints with **zero** findings. Golden files pin three
generated modules (sample, normalize3d, rootsqr). CG-3 (generated RTL passes co-sim
against the golden model — full loop closure) is an integration test that skips
without Verilator and runs in the CI tools job. `pipeforge-cli codegen` emits headlessly.

- Tests: 201 passed, 2 skipped (tool-gated). Coverage ≈89 % (gate: 80 %).
  ruff + mypy --strict(core): clean.
- Known deviations: CG-5 (P1, optional audit-rewrite application behind toggles) is
  not implemented this phase; unsupported DAG constructs (opaque indexing/concat)
  raise a clear CodegenError naming the line.
