# Phase 4 report — SystemVerilog pipeline linter

Built `core/svlint`: a structural model (ports, instances, `PIPE macro uses, assigns)
fed by two backends (SL-1) — pyslang when installed (active here, v11; `PIPE uses are
always extracted textually since they are preprocessor macros) and a documented
regex/structural fallback; the active backend is reported and a simulated-absence test
covers the fallback. Cycle propagation prices every operator and pipe with the same
cost model as the auditor (SL-2): per-instance operand-arrival mismatches report both
signals, their computed stages, and the exact `PIPE fix. SL-3 checks: `_nn` suffix
consistency vs computed cycles, valid-chain delay == data-path delay, reset discipline
(valid FFs reset / data pipes unreset for SRL inference), and `i_<module>_<signal>_<stage>`
instance naming — each individually suppressible (`--disable`). Lint findings carry a
signal key that cross-references DAG nodes when a matching `.m` is loaded (SL-4).
`pipeforge-cli lint` exposes everything headlessly (exit 1 on findings).

Fixture corpus: 10 known-bad files each triggering **exactly** its intended finding
(3× delay-match, 1× suffix, 2× valid-chain, 2× reset, 1× naming, 1× unknown-module)
and 3 known-good files (the README A*B+C example, a scalar multiply-add, a divider
with div_valid chain) that lint clean; both backends agree on the whole corpus.

- Tests: 173 passed. Coverage: core 93 %, overall ≈90 % (gate: 80 %).
  ruff + mypy --strict(core): clean (pyslang imported dynamically; its shipped
  stubs do not parse under mypy).
- Known deviations: none.
